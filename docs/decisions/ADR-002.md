# ADR-002: EF Core Query Strategy

## Status
Accepted

## Context

Transaction list endpoint (`GET /transactions`) ต้องการดึงข้อมูล transactions พร้อม category name/color สำหรับแสดงผลใน UI มีหลายวิธีที่ EF Core รองรับ:

1. **Include (Eager Loading)** — โหลด related entity ทั้งก้อนผ่าน JOIN
2. **Select Projection** — ระบุเฉพาะ columns ที่ต้องการผ่าน SQL SELECT
3. **Lazy Loading** — โหลด related entity เมื่อ access property (ต้องติดตั้ง proxy)

แต่ละวิธีให้ SQL ที่ต่างกัน และมี trade-off ที่ชัดเจน

---

## SQL Comparison (Task 2.13–2.15)

### Pattern 1: Include (Before Refactor)

```csharp
db.Transactions
    .AsNoTracking()
    .Include(t => t.Category)
    .Where(t => t.UserId == userId)
    .ToListAsync()
```

SQL ที่ generate:
```sql
SELECT t.id, t.amount, t.type, t.description, t.transaction_date,
       t.category_id, t.user_id, t.ai_categorized, t.ai_confidence, t.created_at,
       c.id, c.name, c.color, c.icon, c.user_id, c.is_default, c.created_at
FROM transactions AS t
LEFT JOIN categories AS c ON t.category_id = c.id
WHERE t.user_id = @p0
ORDER BY t.transaction_date DESC, t.created_at DESC
```

**ปัญหา:** ดึง Category ทั้งหมด 7 columns ทั้งที่ต้องการแค่ `name` กับ `color`

---

### Pattern 2: Select Projection (After Refactor)

```csharp
db.Transactions
    .AsNoTracking()
    .Where(t => t.UserId == userId)
    .Select(t => new TransactionResponse(
        t.Id, t.Amount, t.Type.ToString(), t.Description, t.TransactionDate,
        t.CategoryId,
        t.Category == null ? null : t.Category.Name,
        t.Category == null ? null : t.Category.Color,
        t.AiCategorized, t.CreatedAt))
    .ToListAsync()
```

SQL ที่ generate:
```sql
SELECT t.id, t.amount, t.type, t.description, t.transaction_date,
       t.category_id, t.ai_categorized, t.created_at,
       c.name, c.color                        -- แค่ 2 columns ที่ใช้จริง
FROM transactions AS t
LEFT JOIN categories AS c ON t.category_id = c.id
WHERE t.user_id = @p0
ORDER BY t.transaction_date DESC, t.created_at DESC
```

**ผลลัพธ์:** โหลดน้อยกว่า — ไม่มี `c.id, c.icon, c.user_id, c.is_default, c.created_at`

### AsNoTracking — ทำไมต้องใส่ (Task 2.14)

EF Core ปกติ **track** ทุก entity ที่โหลดมาใน `ChangeTracker` — เพื่อ detect changes เมื่อ call `SaveChanges()`:

```csharp
// ❌ Without AsNoTracking (default)
var txs = await db.Transactions.Where(...).ToListAsync();
// EF Core เก็บ snapshot ของทุก entity ใน memory → overhead มากขึ้นตาม dataset size
// เมื่อ call SaveChanges() EF Core compare ทุก entity กับ snapshot เพื่อ detect changes
```

```csharp
// ✅ With AsNoTracking
var txs = await db.Transactions.AsNoTracking().Where(...).ToListAsync();
// ไม่มี snapshot → ประหยัด memory + เร็วกว่า สำหรับ read-only operations
```

**กฎ:** ใช้ `AsNoTracking()` เสมอสำหรับ read-only queries (GET endpoints)
**ไม่ใส่เมื่อ:** ต้องการ update entity ในไม่กี่ขั้นถัดไป (เช่น `GetByIdAsync` สำหรับ PUT/DELETE)

## N+1 Problem — สิ่งที่ต้องหลีกเลี่ยง (Task 2.16)

**N+1** คือ anti-pattern ที่เกิดเมื่อโหลด list (1 query) แล้วดึง related data ของแต่ละ item แยก (N queries):

```csharp
// ❌ BAD — N+1 pattern (อย่าทำ)
var txs = await db.Transactions
    .Where(t => t.UserId == userId)
    .ToListAsync();
// SQL 1: SELECT * FROM transactions WHERE user_id = @p0

foreach (var tx in txs)
{
    // EF Core ถ้า enable lazy loading จะยิง query ต่อทุก transaction:
    var categoryName = tx.Category?.Name;
    // SQL 2:   SELECT * FROM categories WHERE id = @categoryId_1
    // SQL 3:   SELECT * FROM categories WHERE id = @categoryId_2
    // ...
    // SQL N+1: SELECT * FROM categories WHERE id = @categoryId_N
}
// ถ้ามี 100 transactions = 101 queries! (แทนที่จะเป็น 1 query)
```

**วิธีแก้ที่ถูกต้องมี 2 แบบ:**

1. **Include** — ดี สำหรับ case ที่ต้องการ related entity ทั้งหมด
   ```csharp
   db.Transactions.Include(t => t.Category).Where(...).ToListAsync()
   // SQL: 1 query + LEFT JOIN
   ```

2. **Select Projection** (preferred) — ดีกว่า สำหรับ case ที่ต้องการแค่บาง fields
   ```csharp
   db.Transactions.Where(...).Select(t => new { ..., Name = t.Category.Name }).ToListAsync()
   // SQL: 1 query + LEFT JOIN + เฉพาะ columns ที่ใช้
   ```

## Decision

ใช้ query pattern ตาม context ดังนี้:

| Scenario | Pattern | เหตุผล |
|----------|---------|--------|
| List queries (GET /transactions) | Select projection + AsNoTracking | โหลดเฉพาะ fields ที่ต้องการ ไม่มี overhead ของ ChangeTracker |
| Single entity reads (GET /transactions/{id}) | Include + AsNoTracking | ต้องการทุก field เพื่อ populate response ครบถ้วน |
| Write operations (POST, PUT, DELETE) | Tracking (ไม่ใส่ AsNoTracking) | ต้องการ ChangeTracker เพื่อ detect changes + update |
| Lazy loading | ❌ ไม่ใช้ | unpredictable N+1, ซ่อน SQL จาก developer |

---

## Architectural Trade-off

การใช้ Select projection ใน repository layer มี trade-off หนึ่ง:

`TransactionRepository` (Infrastructure layer) ใช้ `TransactionResponse` (Application layer DTO) ใน `.Select()` — นี่คือ dependency จาก Infrastructure → Application ที่ทำให้ boundary พร่าเลือนนิดหน่อย

**ทำไมยอมรับได้:**
- Architecture แบบ Layered (ไม่ใช่ Clean/Onion) อนุญาต Infrastructure → Application
- Alternative (return entity แล้ว map ใน Service) ทำให้ต้องโหลด entity ทั้งก้อนก่อน — เสีย benefit ของ projection
- สำหรับ project ขนาดนี้ (personal finance tracker) pragmatic choice นี้คุ้มค่ากว่า complexity ของ CQRS pattern

**ถ้า project ใหญ่ขึ้น:** พิจารณา CQRS — แยก Write Model (entities) กับ Read Model (projections) ออกจากกันอย่างชัดเจน

---

## Alternatives Considered

### Option A: Keep Include, map in Service (ไม่เลือก)
- Service เรียก `repo.GetByUserAsync()` → รับ `IReadOnlyList<Transaction>` → `MapToResponse()`
- ข้อเสีย: โหลด Category entity ทั้งก้อนทุกครั้ง แม้ต้องการแค่ 2 fields

### Option B: CQRS — separate ITransactionQueries interface (ไม่เลือก)
- Read path ใช้ query handler แยก ไม่ผ่าน repository
- ข้อเสีย: over-engineering สำหรับ project ขนาดนี้ — เพิ่ม abstraction โดยไม่จำเป็น

### Option C: Select Projection in Repository (เลือก)
- Refactor `GetByUserAsync` ให้ return `IReadOnlyList<TransactionResponse>` โดยตรง
- ได้ SQL ที่ efficient กว่า + โค้ดยังเข้าใจง่าย

---

## Consequences

**ดีขึ้น:**
- SQL ที่ส่งผ่าน network มี columns น้อยลง — เร็วกว่าเมื่อ dataset ใหญ่
- ไม่มี N+1 risk สำหรับ list endpoint
- ไม่มี ChangeTracker overhead สำหรับ read queries

**ต้องระวัง:**
- `GetByUserAsync` ใน repository รู้จัก `TransactionResponse` DTO — ถ้าเพิ่ม field ใน DTO ต้องอัพเดท Select ด้วย
- `GetByIdAsync` ยังใช้ Include — พฤติกรรมต่างกันสำหรับ single vs list query (documented intentional)
